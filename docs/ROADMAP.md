# Wisp Roadmap

Versioned deliverables for Nephele Wisp. Each version has a clear
kill-point: ship it, collect signal, decide whether the next version
earns its slot. A single-person team cannot afford speculative builds.

## Principles

1. **One platform first; abstract after three.** Do not build a
   multi-platform abstraction layer until at least three platforms
   have concrete content-script implementations. Premature abstraction
   over platform-specific DOM always leaks.
2. **Additive permissions, batched per version.** Chrome Web Store
   review scrutinizes every added permission. Bundle new permissions
   into the version that ships the feature needing them, justified
   per-domain.
3. **Alpha over unpacked, beta over Web Store.** Internal dogfooding
   uses developer-mode unpacked load; public beta only begins after
   Web Store review clears.
4. **Humanization is non-negotiable.** No version bypasses the
   Humanization Pipeline in PROTOCOL.md — not even for debugging.

## v0.4 — Extension-native upload (target: 2026-05)

**Goal**: Replace the Playwright-based B站 uploader with an extension
path that uses the user's real browser. Retire
`tools/publisher/uploaders/*.py` in the Nephele Workshop repo.

**Deliverables**
- [ ] MV3 manifest, minimum permissions: `nativeMessaging`,
      `storage`, `debugger`, `tabs`, `scripting`,
      `host_permissions: ["*://*.bilibili.com/*"]`
- [ ] Service worker: NMH connection, reconnection with exponential
      backoff, heartbeat, message router
- [ ] Content script skeleton with Humanization Pipeline helpers
- [ ] B站 upload flow: image upload → title → topic picker →
      description → save draft (NEVER auto-publish)
- [ ] Nephele side: `main_qt.py --nmh` subcommand, message router,
      asset HTTP server, session token issuance
- [ ] Registration scripts (`nmh/register_win.py`,
      `nmh/register_mac.sh`): write NMH manifest + registry/plist on
      install; remove on uninstall. Both Chrome and Edge.
- [ ] Nephele UI: "Install Wisp Extension" flow deep-linking to
      Web Store and Edge Add-ons
- [ ] Nephele UI: handshake status indicator (connected, disconnected,
      outdated)
- [ ] Fallback UX: if handshake fails, publisher view shows Install
      Wisp CTA + manual copy/paste fallback (NO Playwright resurrection)
- [ ] Chrome Web Store submission (icons, screenshots, privacy
      policy page, per-permission justifications)
- [ ] Edge Add-ons submission (same zip)
- [ ] Delete in Nephele Workshop: `tools/publisher/uploaders/*.py`,
      `tools/publisher/uploader_service.py`,
      `BrowserManager.ensure_user_browser()`, `_find_chrome_exe()`,
      related tests, `scripts/probe_bilibili.py`,
      `scripts/scrape_topic_posts.py`, `scripts/test_bilibili_upload.py`

**Test checklist**
- [ ] Handshake succeeds on fresh Chrome install
- [ ] Handshake succeeds on Edge
- [ ] Upload succeeds on B站 with existing user cookies
- [ ] Upload gracefully fails when user is logged out (`AUTH_REQUIRED`)
- [ ] Captcha detection triggers user notification + 24h cooldown
- [ ] NMH survives Nephele app restart (clean re-handshake)
- [ ] Asset-server token expiration returns 410
- [ ] Uninstalling Nephele removes NMH registry keys / plist
- [ ] Extension works with Chrome window minimized
- [ ] Extension does NOT steal focus at any point during upload

**Kill-point**: if Web Store review rejects twice with unresolvable
demands, pivot to a QR-code login + Playwright-persistent flow (keep
`ensure_user_browser` alive, accept the arms race) and re-evaluate at
v0.5. Do not submit a third time.

## v0.4.1 — Creator dashboard (target: 2026-06)

**Goal**: Read creator-center metrics from B站, 小红书, 抖音, X and
surface a cross-platform dashboard + 48-hour post-publish performance
report.

**Deliverables**
- [ ] Extension: per-platform data fetch helpers (authenticated JSON
      endpoints where available, DOM scraping only as fallback)
- [ ] `host_permissions` additions: `member.bilibili.com`,
      `creator.xiaohongshu.com`, `creator.douyin.com`,
      `x.com/*/analytics`
- [ ] `chrome.alarms` periodic sync (30-minute period with ±5min
      jitter)
- [ ] Local cache at `~/.nephele_workshop/creator_data/<platform>/`
- [ ] Nephele: new `CreatorDashboard` QML view with time-series charts
- [ ] 48-hour report: triggered 48h ±30min after a new post is
      detected, cross-platform aggregated, narrative generated by
      Cloud MAX

**Test checklist**
- [ ] Cold start: first sync completes within 30s per platform
- [ ] Incremental sync: only new data transferred
- [ ] One platform's failure does not break others
- [ ] Data never leaves the user's machine except explicit AI narrative
      invocations
- [ ] 48h report triggers within ±30min of the 48h mark

**Kill-point**: if paywall conversion does not move measurably after
4–6 weeks of dashboard exposure, abandon the creator-analytics
direction. Do not build v0.5 inbox features.

## v0.5 — Inbox & scheduling (target: 2026-07 ~ 2026-08)

**Goal**: Aggregate comments + DMs across platforms, generate reply
drafts (NEVER auto-sent), ship scheduled publishing.

**Deliverables**
- [ ] Extension: comment + DM fetch per platform
- [ ] Extension: `inbox.reply` with strict one-click-sends-once
      semantics (no batching, no scheduling, always user-confirmed
      last mile)
- [ ] `host_permissions` additions: per-platform DM subdomains
- [ ] Nephele: `InboxView` QML with unified inbox across platforms
- [ ] AI reply drafts via Cloud MAX with per-platform voice guidance
      (NEVER auto-sent)
- [ ] Scheduled publish: Nephele-side scheduler issues
      `scheduler.execute` at `target_time ± 15min` jitter

**Kill-point**: if platform rate-limit signals escalate (cooldowns
becoming routine), drop to read-only inbox and shelve write features
until a cleaner route is found.

## v0.6+ — Anti-piracy & commission CRM

## v1.0 — Growth coach (target: 2026 Q4)

Narrative AI reports, growth funnel analytics, post-type effectiveness
attribution. Scope defined closer to kickoff.

## Non-goals (permanent)

These are out of scope regardless of demand:

- Auto-like / auto-follow / auto-unfollow / auto-comment
- Multi-account switching within the same browser profile
- Follower-count manipulation
- Bulk scraping of other users' content beyond what the user could
  see by browsing normally
- Bypassing platform captchas (we pause; we don't solve)

These are red lines. Violating them turns Wisp into a gray-market
growth-hack tool rather than a legitimate artist aid, and destroys
the trust premise that justifies asking users to install a browser
extension at all.
